# Pullminder Registry

The official open-source registry of rule packs for [Pullminder](https://pullminder.com). These packs define the detection patterns and policy rules that Pullminder evaluates during pull request reviews.

Packs work offline with the CLI — no platform account required.

## Quick start

```bash
# Install the CLI
brew install pullminder/tap/pullminder
# or: npm install -g pullminder

# See available packs
pullminder packs list

# Run all enabled packs against your current branch
pullminder check

# Enable a specific pack
pullminder packs enable go-security
```

## Available packs

Pattern-matching rules that scan code for security issues, leaked credentials, and risky patterns.

|------|------|-----------|-------------|---------|

| `rust-security` | Rust Security | Rust | Unsafe blocks, memory safety issues | No |
| `ruby-security` | Ruby Security | Ruby | Mass assignment, command injection in Rails | No |
| `php-security` | PHP Security | PHP | SQL injection, file inclusion, deserialization | No |
| `react-security` | React Security | JS/TS | XSS vectors, unsafe innerHTML, prototype pollution | No |
| `java-security` | Java Security | Java | SQL injection, XXE, deserialization, SSRF | No |
| `csharp-security` | C# Security | C# | SqlCommand injection, BinaryFormatter, XXE | No |
| `kotlin-security` | Kotlin Security | Kotlin | WebView, SharedPreferences, exported components | No |
| `swift-security` | Swift Security | Swift | ATS bypass, keychain access, biometric auth | No |
| `shell-security` | Shell Security | Shell | Eval injection, curl-pipe, chmod, temp files | No |
| `infra-security` | Infrastructure Security | All | Dockerfile, Kubernetes, Terraform, CI/CD misconfigurations | Yes |
| `crypto-anti-patterns` | Cryptographic Anti-Patterns | All | Weak crypto detection (MD5, DES, ECB, small keys) | No |

### Governance packs

Detection rules for code provenance, process enforcement, and review quality.

| Slug | Name | Description | Default |
|------|------|-------------|---------|
| `bot-detection` | Bot Detection | PRs authored by bots or automated tools | Yes |
| `ai-detection` | AI Detection | Code likely generated by AI assistants | Yes |
| `ai-senior-review` | AI Senior Review | Requires senior approval for high-risk AI-generated PRs | Yes |
| `dependency-detection` | Dependency Detection | Lockfile and manifest changes flagged for review | Yes |

### Compliance packs

Privacy, licensing, and standards-mapping rules.

| Slug | Name | Description | Default |
|------|------|-------------|---------|
| `pii-leakage` | PII Leakage Detection | Detects PII (SSN, credit cards, emails) in logging contexts | No |

## Managing packs

```bash
# List all packs and their status
pullminder packs list

# Show details about a pack (rule count, severities)
pullminder packs info secrets

# Enable or disable a pack
pullminder packs enable rust-security
pullminder packs disable bot-detection

# JSON output (useful for scripting and AI agents)
pullminder packs list --json
```

## Custom registries

You can create your own private registry with custom rule packs for your team:

```bash
# Scaffold a new registry
pullminder registry init my-org-rules

# Add a pack
pullminder registry pack add --slug our-standards --kind policy --name "Our Standards"

# Validate the registry
pullminder registry validate --strict
```

Custom registries follow the same pack.yaml schema as this public registry. See the [pack structure](#pack-structure) below.

## For pack authors

### Pack structure

Each pack lives in `packs/<slug>/` and contains a `pack.yaml` file:

```yaml
slug: my-pack
name: My Pack
kind: detection          # "detection" or "policy"
action: flag             # "flag", "block", or "warn"
version: 1.0.0
schema_version: 1
author: your-github-handle
max_weight: 10

scoring:
  model: additive
  base_weight: 5

patterns:
  - name: Example Pattern
    rule_id: MY-001
    regex: "pattern-to-match"
    language: "*"          # "*" for all, or "go", "python", etc.
    severity: high         # critical, high, medium, low, info
    category: security
    fix_templates:
      - "Suggested fix description"

overrides:
  ignore_paths:
    - "**/testdata/**"
```

### Testing packs

Add test fixtures alongside your pack, then run:

```bash
# Test all packs
pullminder rules test

# Test a specific pack
pullminder rules test --pack my-pack --verbose
```

### Publishing to this registry

Submit your pack via fork-and-PR:

```bash
# Validate and open a PR automatically
pullminder rules publish --pack my-pack

# Dry run (validate only, no PR)
pullminder rules publish --pack my-pack --dry-run
```

See [CONTRIBUTING.md](CONTRIBUTING.md) for the full authoring and submission guide.

## AI agent integration

Every CLI command supports the `--agent` flag for structured JSON output, making packs queryable by AI coding agents (Copilot, Claude Code, Cursor, etc.):

```bash
pullminder packs list --json
pullminder check --agent
```
